0. Purpose:

               The purpose for this document is to include exactly the questions such that, if each were 
               answered by a client representative who is knowledgeable about the desired application, 
               we would know everything necessary to write a requirements document for the software 
               development project and to understand the project. 

               Interview(when) : February 10th, 2016
               Interview(where): VH 1328
               Interview(who)  : Professor Jon Beck
               Interview(What) : Asking questions about a computer system that can normalize mailing addresses

1. Identify actors:

  • 1. Who are the end users?
    • a. Will the end user be a person?
    • b. Will the end user be a computer system?
    • c. Will the end user be a certain department within a company?
  • 2. Will this system be a standalone system or act as a plugin or widget that can interact with other systems?

2. Identify scenarios:

  • 1. How will this program be initiated by a user?
  • 2. If an authorized user fails to enter the correct credential, how many incorrect attempts should be allowed?
  • 3. Should the system standardize an invalid address (e.g. address that doesn’t exist) or try to correct that invalid address into a valid address?
  • 4. Should the system fix a mailing address that doesn’t follow the standard rules (e.g. put the zip-code in the front)?
  • 5. Should the system try to complete a mailing address when there is no enough information (e.g. an address that doesn’t have city, state, and zip-code)?
  • 6. What will happen if the system unexpectedly terminates in the middle of a normalization process?
  • 7. How will this program be terminated by a user?
    • a. Would user logging out terminate the system process?

3. Identify use cases:

  • 1. Should the system require users to login in order to use the system?
  • 2. Should the system fix the data if the input mailing address is invalid?
  • 3. Should the system autocomplete the data if the input mailing address is incomplete?
  • 4. Should the system format the mailing addresses after the standardization process?
  • 5. Should the system notify users once the standardization process is completed?
  • 6. Should the system allow users to pause the standardization process?
  • 7. Should the system have a restore state that it can revert to after an unexpected shutdown?

4. Identify non-functional requirements:

  • 1. What are the acceptable system environments?
    • a. Should this system be a web, desktop, or mobile application?
  • 2. What are the performance requirements for this system? How fast must the system standardize a given set of mailing addresses?
  • 3. Should the system be able to interact with maps?
  • 4. What types of security features are required for the data transmission process?
    • a. Should the system encrypt the data before it is transmitted?
  • 5. Should the system be able to perform tasks both online and offline?

5. Identify functional requirements:

  • 1. Should the system sort the standardized mailing addresses?
  • 2. Should the system accept variety of input types (e.g. a database, a text file, input files, etc.)?
  • 3. Should the system format the output result in a particular way (e.g. label)?
  • 4. Should the system be able to send a notification email to users after the completion of standardization process?
  • 5. Should the system store the invalid data set?
